# 产品需求文档 (PRD)

## 1. 项目概述
- **项目名称**: Nova-RAG (企业级智能知识库系统)
- **版本**: V1.0 (MVP 最小可行性产品)
- **核心定位**: 基于大语言模型（LLM）与检索增强生成（RAG）技术的企业专属知识问答平台。
- **业务价值**: 解决企业内部海量文档检索难、阅读成本高的问题，实现“一问即知，并附来源”的 AI 原生级办公体验。

## 2. 目标用户与使用场景
- **目标用户**: 企业内部全体员工（MVP 阶段暂定不区分多租户权限）。
- **典型场景**: 
  - 新员工入职，查询冗长的《员工手册》和报销流程。
  - 研发人员遇到特定技术问题，直接提问并获取内部技术方案的代码片段及原文档出处。

## 3. 核心功能需求 (Functional Requirements)

### 3.1 知识库管理模块 (Knowledge Base Management)
- **文档上传支持**: 支持 `.pdf`, `.md`, `.docx` 格式文档的单文件及批量上传。
- **解析与存储状态追踪**: 
  - 前端需提供可视化的解析进度反馈（如：上传中 -> 切片解析中 -> 向量化完成）。
  - 支持对已上传文档的列表查看与一键删除。

### 3.2 AI 检索与处理引擎 (RAG Engine)
- **核心大模型选型**: Minimax m2.7（确保国内合规，中文语境表现优异）。
- **智能分块算法 (Chunking Strategy)**:
  - 必须采用 **父子文档检索 (Parent-Child Retrieval)** 技术。
  - **规则**: 将文档切分为较小的 Child 块（用于提高向量检索的匹配命中率），但最终提交给 Minimax 大模型的是对应的 Parent 块（用于保留完整的上下文语义，杜绝 AI 断章取义）。
- **防幻觉机制 (Anti-Hallucination)**: 系统必须在 Prompt 层面强制要求大模型“严格依据检索到的文本作答，不得伪造信息；若检索内容无相关答案，需明确告知用户未找到”。

### 3.3 智能对话交互界面 (Chat Interface)
- **流式输出 (Streaming)**: 采用 Server-Sent Events (SSE)，实现类似 ChatGPT 的打字机逐字输出效果，降低用户首字等待焦虑。
- **富文本与代码高亮**: 支持基于 Markdown 语法的复杂格式渲染，包括加粗、表格、代码块及语法高亮。
- **来源溯源展示 (Source Attribution)**:
  - 每次 AI 回答结束后，必须在末尾清晰列出引用的文档名称及对应片段。
  - **UI/UX 强制约束**: 溯源展示必须使用自定义样式的卡片或普通段落超链接。**前端渲染逻辑中，绝对禁止使用 `<cite>` 标签或相关自定义 cite 语法块**，以防止底层 Markdown 解析器抛出致命报错或导致页面崩溃。
- **多轮对话记忆**: 系统需支持带上下文的连续追问，历史记录需被妥善持久化。

## 4. 非功能性需求 (Non-Functional Requirements)
- **性能指标**: 文档检索到大模型吐出首个字符的响应时间（TTFB）需控制在 1.5 秒以内。
- **合规与安全性**: 数据不离境，API 交互必须走企业内网或加密代理传输。
- **可维护性**: 业务侧需保留完整的运行日志（特别是大模型的出入参记录），以便后续进行 Prompt 调优。
