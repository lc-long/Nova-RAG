## 一、环境篇 — Windows CGO 编译与 SQLite 替换

### 问题
在 Windows 环境下，后端 Go 服务使用 `modernc.org/sqlite`（纯 Go 实现）替换了原生 `CGO` + `mattn/go-sqlite3`，解决了跨平台编译问题。

### 关键决策
- 选择 `modernc.org/sqlite` 而非 CGO绑定：无需 gcc/mingw 环境
- 数据库文件：`backend/lumina.db`

### 注意事项
- V2.0 如需使用其他 C 绑定库（如某些机器学习绑定），需重新评估 CGO 策略

## 二、通信篇 — Go 异步调用 Python 的 404 / 路径断层

### 问题
Go 后端通过 HTTP 调用 Python（FastAPI）时出现两类路径问题：

1. **文件路径断层**：Go 将文件上传到 `backend/uploads/`，但传给 Python 的是相对路径 `uploads/xxx.pdf`，Python 进程的工作目录与 Go 不同，导致 Python 侧 `os.path.exists()` 返回 `False`，404。
2. **异步入库竞态**：Go `Upload` handler 异步调用 Python `/ingest`，但响应立即返回 200，Python 实际还未完成入库，前端查询返回空。

### 修复方案
- **绝对路径硬编码**：Go → Python 传递绝对路径 `D:\code\ai\rag\backend\uploads\xxx.pdf`
- **IP 硬编码**：`127.0.0.1:5000` 而非 `localhost:5000`（Windows localhost 解析有时延）
- **入库轮询**：前端/测试脚本轮询 ChromaDB `collection.count()` 确认数据就绪后再查询

### 关键代码
```go
// backend/internal/handler/docs.go
absPath := "D:\\code\\ai\\rag\\backend\\uploads\\" + docID + "_" + header.Filename
url := fmt.Sprintf("http://127.0.0.1:%s/ingest", h.pythonPort)
```

## 三、协议篇 — SSE 流式 UI 乱码与双头像 Bug

### 问题
V1.0 早期实现中，Python MiniMax 客户端将 `reasoning_content` 和 `content` 拼接成单个字符串返回给 Go，Go 再直接转发给前端。前端用字符串匹配解析 `<think>` 标签，导致：

1. **重影问题**：推理内容与回答内容在 UI 层交错渲染
2. **双头像**：消息 ID 重复或状态更新时机错误导致两条 assistant 消息
3. **标签外溢**：`<think>` 标签未正确闭合时，Markdown 渲染混乱

### 根因
MiniMax V2 API 的 SSE delta 有两个独立字段：
- `delta.reasoning_content` → AI 推理过程
- `delta.content` → 最终回答

早期实现把这两个字段拼接成 `<think>+</think>` 字符串，污染了回答文本。

### 修复方案：type 双轨制 JSON 协议
不再在文本层打标签，而是在 JSON 协议层分离：

```json
// Python → Go → 前端
{"type": "reasoning", "content": "用户询问的是..."}
{"type": "answer", "content": "根据上下文，答案是..."}
{"type": "done", "references": [...]}
```

### 关键改动文件
- `python/src/core/llm/minimax.py`：直接 yield `StreamChunk(chunk_type="reasoning"|"answer"|"done")`
- `python/src/api/server.py`：`generate()` 根据 `chunk.chunk_type` 输出类型化 JSON
- `frontend/src/components/ChatArea.tsx`：`Message` 接口改为分离 `reasoning` / `content` 字段，UI  Accordion 渲染

## 四、RAG 篇 — 指令型检索失效与元数据注入

### 问题
当用户问"总结腾讯 2023 年报中关于 AI 的战略规划"时，检索返回空或无关内容。原因：

1. **向量相似度不匹配**：查询词"总结"、"AI 战略"与 Chunk 中的"董事会报告"、"财务数据"向量距离远
2. **摘要类 query 天然难度高**：需要跨多个 chunk 聚合信息

### 修复方案：来源文件元数据前缀注入
在 `python/src/api/server.py` 的 `/ingest` endpoint 中，对每个 Chunk 的 content 头部注入：

```python
prefix = f"[来源文件：{req.filename}]\n"
chunk.content = prefix + chunk.content
```

这样检索时，问"腾讯年报"可以直接命中带 `[来源文件：synthetic_financial_report.txt]` 前缀的 chunk。

### 关键改动
- `ChromaRetriever` 增加 `distance_threshold=10.0` 过滤 + 兜底无阈值查询
- metadata 增加 `source` 字段支持文件名精确匹配

## 五、PDF 解析篇 — pypdf/pdfplumber 的局限

### 问题
实测发现，`pdfplumber` 在以下场景表现不佳：
- **扫描版 PDF**（DJI Mavic 3 用户手册）：表格文本被当成普通文本流处理
- **多列表格**：参数表（最大速度、飞行时间等）被拆散成不相关的碎片 Chunk
- **复杂版式**：中文竖排、多栏布局会导致段落顺序错乱

### V2.0 改进方案：PyMuPDF 表格检测 + pdfplumber 文本（HOLD）

**最初方案**：直接用 PyMuPDF 替换 pdfplumber（`page.get_text("text")`）

**遇到的问题**：PyMuPDF 在 DJI Mavic 3 手册上提取的中文文本全部乱码：
```
ly App ֧��Android 6.0 ������ϵͳ��֧��iOS 11.0 ������ϵͳ��
```
这是因为 PyMuPDF 的 `get_text("text")` 对嵌入字体的 CJK 字符没有做 unicode 映射，
而 pdfplumber 内置了 CJK 字体解码器。

**最终方案（hybrid）**：两者结合
```
pdfplumber  → 提取可靠的中文文本
PyMuPDF    → page.find_tables() 做表格检测
           → 表格转 Markdown 后注入 pdfplumber 文本
```
这样既有可靠的 CJK 文本，又有 PyMuPDF 的强大表格检测能力。

## 六、ChromaDB Reset 的 Collection ID 陷阱

### 问题
调用 `/reset_db`（删除 collection）后，server 进程持有旧的 collection 对象引用（`vector_store._collection`），后续 `/ingest` 和 `/process_query` 请求使用该失效引用，导致：

```
chromadb.errors.NotFoundError: Collection [old-uuid] does not exist
```

### 修复
`server.py` 的 `reset_db` endpoint 中，将 `_collection = None`（而非重建），让下次访问时惰性重建：

```python
vector_store.client.delete_collection(name=vector_store.collection_name)
vector_store._collection = None  # 惰性重建
```

### 问题
早期使用 `api.minimax.com`（错误域名）和模型名 `MiniMax-M2`（不可用）。

- **Embedder 模型**：从 `all-MiniLM-L6-v2` 升级为 `paraphrase-multilingual-MiniLM-L12-v2`（支持中文语义）

## 八、V1.0 RAG 核心策略基线（Baseline）

> 以下参数和策略为 V1.0 MVP 的固定基线，是后续迭代的基准。任何偏离以下基线的改动都应记录在 v2.0_ROADMAP.md 中。

### 8.1 文档处理策略（Document Processing）

#### 解析器

| 文件类型 | 解析库 | 备注 |
|---------|--------|------|
| PDF | `pdfplumber` | 当前主力解析库 |
| DOCX | `python-docx` | 通过 `docx_parser.py` 调用 |

PDF 解析逻辑（`pdfplumber`）：
```python
# python/src/core/chunker/pdf_parser.py
with pdfplumber.open(file_path) as pdf:
    for page_num, page in enumerate(pdf.pages, start=1):
        text = page.extract_text() or ""
        if text.strip():
            yield text, page_num, text
```
纯文本流提取，按页返回。**表格结构**在 V1.0 中未做专门保留，会被当作普通文本处理（这是 V2.0 的升级点）。

#### 切片算法（Chunking）

**类型**：Parent-Child 双层切分

| 参数 | 值 | 说明 |
|------|-----|------|
| `parent_chunk_size` | 2000 字符 | 上层语义块，保留完整段落 |
| `child_chunk_size` | 500 字符 | 下层细粒度块，提高检索命中率 |
| `overlap` | 50 字符 | child 层重叠，防止截断丢失语义 |
| `separators` | `["\n\n", "\n", "。", "！", "？", ". ", " ", ""]` | 递归按标点切分 |

切分流程：
```
原始文本
  → 按 2000 字符切 parent chunk
  → 每个 parent 再切 500 字符 child chunk（带 50 字符重叠）
  → 存储时建立 parent_id 关联
  → 检索时优先命中 child，再通过 parent_id 还原完整上下文
```

#### 元数据清洗（Metadata Cleaning）

**来源文件前缀注入**（入库时强制注入）：
```python
# python/src/api/server.py → /ingest endpoint
prefix = f"[来源文件：{req.filename}]\n"
for chunk in chunks:
    chunk.content = prefix + chunk.content
```

ChromaDB metadata 结构：
```json
{
  "doc_id": "xxx",
  "source": "byd_2023_annual_report.pdf",  // 文件名，用于精确过滤
  "chunk_type": "parent" | "child",
  "parent_id": "xxx",                         // child chunk 指向 parent
  "page_number": 0
}
```

### 8.2 检索召回策略（Retrieval）

#### 向量模型（Embedding Model）

| 属性 | 值 |
|------|-----|
| 模型名 | `paraphrase-multilingual-MiniLM-L12-v2` |
| 维度 | 384 维 |
| 来源 | HuggingFace Sentence Transformers |
| 用途 | 中英双语语义嵌入，支持中文语义匹配 |
| 加载方式 | 单例模式（`SentenceTransformerEmbedder.get_instance()`） |

#### 召回数量（Top-K）

| 参数 | 值 | 说明 |
|------|-----|------|
| `top_k`（向量检索） | 5 | 从 ChromaDB 召回 5 个最近邻 chunk |
| `top_k`（最终入模型） | 5 | 取回 5 个 chunk 去重后喂给 MiniMax |

#### 过滤与阈值（Threshold & Filtering）

| 参数 | 值 | 说明 |
|------|-----|------|
| `distance_threshold` | 10.0 | 向量距离超过 10.0 的结果被丢弃 |
| 降级策略 | 无阈值兜底 | 如果过滤后结果为空，则回退到 top-5 **不带距离过滤**的查询 |

去重规则：
- 同一 `parent_id` 的 child chunk 只保留第一个
- 相同文本（去换行后）的 chunk 只保留一个

#### 上下文组装（Context Prompting）

最终喂给 MiniMax 的 Prompt 结构：
```
你是一个企业知识库助手。请严格根据提供的上下文信息回答用户问题。
如果上下文中没有相关信息，请明确告知用户"抱歉，我在知识库中未找到相关内容"。

## 上下文信息
[1] Document: {doc_id}
{parent_content}

[2] Document: {doc_id}
{parent_content}
...

## 对话历史
user: {content}
assistant:

## 用户问题
```

其中 `[N] Document` 部分由 `ChromaRetriever._build_context_prompt()` 生成，`_build_references()` 为最终 SSE 响应提供 `[N]` 格式的参考来源列表。
